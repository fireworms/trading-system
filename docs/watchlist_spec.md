## 작업 전 먼저 할 것
기존 코드베이스를 먼저 파악해. 현재 FastAPI 백엔드 구조(라우터/모델/DB 세션), PostgreSQL 스키마, KIS API 클라이언트, Gemini 호출 모듈, Next.js 탭 구조와 컴포넌트 패턴을 확인하고 거기에 **일관되게** 통합해. 아래는 스펙이지 새 프로젝트가 아님 — 기존 네이밍/폴더 구조/에러 핸들링 컨벤션을 따를 것.

## 목적
자동매매 탭과 별개로, **중장기 관점 수동매매**용 관심종목 분석 탭. 핵심은 AI 예측이 아니라 **매매 일지** — 종목별·일자별로 "그날 내가 왜 이렇게 봤는지"를 데이터와 함께 타임스탬프로 박아두고 나중에 검증 가능하게 하는 것.

## 핵심 설계 원칙 (중요 — 이거 어기면 잘못 만든 거)
1. **AI 역할 = 데이터 집계 + 생각 구조화, 예측 아님.** Gemini한테 "오를까 내릴까" 시키지 말고, 흩어진 지표·수급·뉴스를 모아서 판단 가능한 구조로 정리하게 시킴. 출력에 근거 없는 방향성 단언("단기 조정 후 상승") 금지.
2. **분석 트리거는 이벤트 기반 + 수동.** 매일 자동 분석 안 함(중장기인데 EPS/PER은 분기마다 바뀌어서 매일 돌리면 입력이 거의 안 변해 노이즈만 쌓임). 트리거: 실적발표 / 주요공시 / 수급 급변 / 주가 급변 + 사용자가 누를 때.
3. **분석할 때 입력 데이터 스냅샷도 같이 저장.** 그날 쓴 지표값 + 뉴스 출처(URL/제목)를 분석 결과와 함께 DB에 저장. 안 그러면 6개월 뒤에 AI가 뭘 보고 그렇게 판단했는지 재구성 불가 = 일지 가치 반토막.
4. **recency bias 억제.** Gemini가 최신 뉴스에 과반응하는 거 중장기엔 버그. 프롬프트에서 "펀더멘털 우선, 뉴스는 보조 + 일시적 노이즈냐 구조적 변화냐 구분"을 명시적으로 강제.

## 데이터 모델 (기존 스키마 스타일에 맞춰 조정)
- `watchlist_stocks`: ticker, name, sector(nullable), memo, added_at
- `stock_analyses`:
  - id, ticker, analysis_date, created_at, gemini_model
  - 구조화 출력 필드 (아래 "출력 구조" 참조) — JSONB 컬럼 하나로 묶어도 됨
  - `input_snapshot` (JSONB): 그날 KIS 지표 raw + 수급 데이터 + 사용된 뉴스 출처 리스트
  - `raw_response` (Gemini 원문, 디버깅용)
- (선택) `analysis_triggers`: 어떤 이벤트로 분석이 돌았는지 기록 (trigger_type)

## API 엔드포인트
- 관심종목 CRUD: 추가 / 삭제 / 목록 / 메모 수정
- `POST /analyze` (ticker, date, trigger_type): KIS 6개월 지표 수집 → 아래 분석 입력 데이터 수집 → Gemini 호출 → 스냅샷+결과 저장 → 반환
- `GET /analyses/{ticker}`: 일자별 분석 이력 (리스트)
- `GET /analyses/{id}` (또는 ticker+date): 단일 분석 상세

## 분석 입력 데이터 — 우선순위순
KIS API로 실제 뭐가 오는지 먼저 확인하고, **없는 건 가짜 엔드포인트로 채우지 말고 "데이터 없음"으로 플래그** 처리해. 컨센서스·일부 펀더멘털은 KIS에 없을 가능성 큼.

1. **펀더멘털 심화** (6개월 PER보다 중요): 분기 매출/영업이익/순이익 YoY 추세, 영업이익률, ROE, 부채비율, PER/PBR 밴드(자기 과거 및 동종업종 대비)
2. **컨센서스**: 증권사 목표주가, 실적 추정치 (forward — trailing 지표 보완). KIS에 없으면 별도 소스 필요하다고 플래그
3. **수급** (한국시장 핵심): 외국인/기관 3~6개월 누적 순매수 추세, 외국인 보유비중 변화. KIS `inquire-investor` (주식현재가 투자자) 엔드포인트로 일별 개인/외국인/기관 순매수 수량+금액 조회 가능. ※ 기존 수급 분석 모듈 있으면 재사용
4. **공시/뉴스** (Gemini 검색): DART 공시(실적/유증/자사주/대형계약/합병), 섹터·테마 동향, 정책 모멘텀
5. **매크로 (섹터 한정)**: 환율(수출주), 금리, 유가·원자재(해당 섹터), 코스피/코스닥 국면
6. **리스크 플래그**: 관리종목/거래정지 리스크, 최대주주 지분 변동, 소송/규제

## Gemini 분석 프롬프트 설계
시스템 프롬프트에 명시:
- 역할: 데이터를 구조화하는 애널리스트. 주가 방향 예측 금지.
- 입력: 위 지표 + 수급 + 뉴스
- 펀더멘털 우선, 뉴스 보조. 뉴스는 "일시적 노이즈 vs 구조적 변화"로 구분해 다룰 것.
- 출력은 아래 구조의 JSON으로만 (preamble/markdown 백틱 없이 → 파싱 안정성)

**출력 구조 (JSON 필드):**
- `논거`: 현재 상태 요약 (데이터 기반)
- `단기_촉매`: 단기 이벤트 + 예상 시점 (있으면)
- `장기_논거`: 중장기 투자 논거
- `무효화_조건`: "이 신호 뜨면 내 판단이 틀린 거다" — falsifiable하게. ※ 제일 중요, 절대 빼먹지 말 것
- `밸류_코멘트`: 현재 밸류에이션 평가 (밴드 대비 싼지/비싼지)
